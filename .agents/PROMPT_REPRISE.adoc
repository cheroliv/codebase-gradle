== 🔑 Règle EAGER — Ségrégation Vision/Opinion/Stratégie (rappel de AGENT_GOVERNANCE Règle 2bis)

Le LLM EST le premier responsable de classifier chaque information avant dilution :
- VISION → publiable (documents racine + blog cheroliv.com)
- OPINION → CONFINÉE Cercle 0 (jamais diluée, jamais publiée)
- STRATÉGIE → dilution restreinte (docs racine section business, PAS de blog)
Cas ambigus → remonter à l'humain.

== Mission — STIMULUS : Ségrégation Vision/Opinion en Base de Données, Pas dans les Documents

=== Le constat

Actuellement, la classification Vision/Opinion est pensée comme un **marquage
dans les documents** (`// @vision`, `// @opinion` en commentaires AsciiDoc, tags
en tête de section). Ce marquage pollue le contexte — il ajoute du bruit
dans les fichiers que le LLM lit en EAGER/LAZY.

=== L'idée

La classification Vision/Opinion doit être **externalisée du document**.
Le document reste propre. La métadonnée de classification vit dans une base
de données relationnelle (PostgreSQL) — une table `epistemic_classification`
qui référence :
- Le chemin du fichier (ou la section)
- Le statut (VISION, OPINION, STRATEGIE, HYPOTHESE)
- La date de classification
- Le classifieur (LLM ou humain)
- Le niveau de confiance

Le `ClassificationGraph` LangGraph4j (EPIC 13) lit cette table avant de router.
Le LLM lit le document sans aucun marquage — il interroge la table pour savoir
ce qui est diluable.

=== Questions à challenger en session 1

. *Quelle granularité de référencement ?* Fichier entier ? Section ?
  Paragraphe ? Ligne ? Un fichier peut contenir à la fois de la Vision et de
  l'Opinion — comment granulariser ?

. *Qui peuple la table ?* Le LLM (via ClassifierNode) ? L'humain ?
  Un processus hybride (LLM propose, humain valide) ?

. *Quand a lieu la classification ?* À la création du contenu ? En fin de
  session ? Les deux ?

. *Comment la table est-elle interrogée par le LLM ?* Via une recherche
  SQL directe ? Via pgvector (similarité avec la section en cours) ?
  Via un appel d'outil LangChain4j ?

. *Est-ce que la table remplace complètement le marquage document ?* Ou
  est-ce un cache complémentaire ? La table est la source de vérité,
  le document n'est jamais marqué.

=== Prototype d'implantation

[source]
----
PostgreSQL table epistemic_classification:
  - id                  UUID
  - file_path           TEXT     (chemin relatif workspace)
  - section_identifier  TEXT     (titre de section, anchor AsciiDoc, ou hash du contenu)
  - content_hash        TEXT     (SHA256 du contenu — pour détection de delta)
  - status              ENUM     (VISION, OPINION, STRATEGIE, HYPOTHESE)
  - classifier          ENUM     (LLM, HUMAN)
  - confidence          FLOAT    (0.0 → 1.0)
  - classified_at       TIMESTAMP
  - expires_at          TIMESTAMP (NULL = permanent, sinon maturation programmée)
  - parent_id           UUID     (auto-ref: trace la maturation Opinion → Vision)
----
