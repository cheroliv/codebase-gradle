== Contexte

Suite à l'ajout du feature `9_3_4_workspace_walker_adoc.feature`, 7 définitions de
steps Cucumber étaient en doublon entre `DatasetEmbeddingSteps` et `WorkspaceWalkerSteps`
(`a pgvector container is running`, `I compute embeddings for all chunks`,
`every chunk has a non-null embedding...`, `I query with the phrase {string}`,
`I insert the documents and chunks...`, `the documents table has exactly {int} rows`,
`the chunks table has more than 0 rows`). L'état (conteneur, fileChunks, model,
topResults) était isolé dans chaque classe → comportement indéfini.

== Actions

. Création `PgVectorTestContext.kt` — singleton partageant :
  * Conteneur pgvector (`startContainer()`, `jdbcUrl()`, `jdbcUser()`, `jdbcPassword()`)
  * Modèle ONNX `AllMiniLmL6V2EmbeddingModel` (lazy init)
  * `fileChunks: MutableMap`
  * `topResults: List<TopResult>`
  * Méthodes de chunking (`splitIntoSentenceLevelChunks`, `estimateTokenCount`, etc.)
. Refactoring `DatasetEmbeddingSteps.kt` → conserve toutes les steps partagées comme
  canon, délègue l'état à `PgVectorTestContext`
. Refactoring `WorkspaceWalkerSteps.kt` → supprime les 7 steps dupliqués + l'état isolé,
  garde uniquement ses 6 steps spécifiques (walk, extension check, vérification type
  adoc/kt)

== Leçon

Ne jamais dupliquer un état de container Testcontainers entre plusieurs classes
Cucumber. Extraire dans un singleton `object` partagé. La query step canonique
doit inclure `d.file_name` dans le SELECT (pas seulement `c.chunk_text`) pour
permettre la vérification du type de fichier source (adoc vs kt).
