== Atelier 1 : Vision

=== QQOQCP

[cols="2,5"]
|===
| Qui | opencode (l'agent CLI) + le dev humain qui ouvre une session dans un borough
| Quoi | Un fichier de contexte injectable par borough, généré à partir des sources indexées en pgvector + fichiers EAGER + Knowledge Graph workspace
| Où | Dans `/tmp/opencode-context.txt` (branché sur `opencode.json` racine workspace)
| Quand | Avant chaque session opencode. Pipeline déclenché manuellement ou automatiquement via engine N3
| Comment | Tâche Gradle `prepareContext` dans codebase-plugin N1. Appelé par borough N2 ou par engine N3 (`assembleCompositeContext`)
| Pourquoi | Réduire les 20k-30k tokens EAGER par session. Substituer le chargement exhaustif de tous les fichiers gouvernance par un résumé ciblé RAG+EAGER+Graphify
|===

=== Product Vision Board

==== Target Group
- Agent opencode (consommateur primaire)
- Développeur humain ouvrant une session dans un borough N2
- Engine N3 (assembleWorkspaceContext pour contexte global)

==== Needs (5 besoins principaux)
. Comprendre le statut du borough (EPICs, roadmap, backlog) sans charger tous les fichiers EAGER
. Obtenir le contexte RAG pertinent sur une question métier (pgvector cosine similarity)
. Visualiser la position du borough dans le graphe de dépendances workspace
. Résilience : fonctionner même si pgvector est down (fallback EAGER+Graphify)
. Pipeline automatisable (appelé par engine N3, CI/CD)

==== Product (proposition de valeur)
Un plugin Gradle N1 qui, pour un borough donné, produit un fichier de contexte
structuré (EAGER + RAG + Graphify) injectable dans opencode. Remplace le chargement
manuel de 3-5 fichiers EAGER par borough.

==== 5 fonctionnalités principales
. `prepareContext` — scan + index + RAG + inject par borough
. `CompositeContextBuilder.buildScoped()` — scope à un seul borough
. `OpencodeInjector.injectToFile()` — formatage opencode-compatible
. Résilience RAG (pgvector down → fallback message)
. `assembleWorkspaceContext` — agrégation cross-borough (engine N3 ou codebase N1)

==== Business Goal (critères de succès)
- 2430 bytes de contexte généré pour codex-gradle ✅ (session 029)
- Réduction de 50%+ des tokens EAGER par session (cible : <8k tokens au lieu de 20k-30k)
- 100% des boroughs N2 couverts (12 boroughs)
- Tests Cucumber passants pour `prepareContext` et `assembleWorkspaceContext`

==== Vision (1 phrase)
Un pipeline de contexte augmenté par borough qui nourrit opencode avec
un résumé EAGER ciblé + du RAG pgvector pertinent + une carte Knowledge Graph,
en remplacement du chargement exhaustif des fichiers de gouvernance.

== Atelier 2 : Personas / Parcours utilisateurs

=== Persona 1 : Agent opencode (terminal consommateur)
- *Cible* : Agent opencode qui démarre une session
- *But* : Recevoir le contexte minimum pertinent pour le borough cible sans charger 10+ fichiers EAGER
- *Parcours utilisateur* :
  . Le dev ouvre une session opencode dans un borough (ex: codex-gradle)
  . opencode.json racine charge `/tmp/opencode-context.txt`
  . Le fichier contient : INDEX.adoc du borough (EAGER tronqué) + top-K RAG sur "architecture du workspace" + stats Graphify
  . L'agent opencode lit ce fichier en 1 bloc → ~2-3k tokens au lieu de 20k-30k

=== Persona 2 : Développeur humain
- *Cible* : Le dev qui prépare une session de travail
- *But* : Générer le contexte augmenté pour le borough sur lequel il va travailler
- *Parcours utilisateur* :
  . `cd foundry/public/codex-gradle && ./gradlew prepareContext`
  . La tâche scanne les sources, indexe dans pgvector, build le contexte scoped
  . Résultat : `build/context/codex-gradle.context.txt` (local)
  . Optionnel : copier vers `/tmp/opencode-context.txt` ou laisser engine N3 le faire

=== Persona 3 : Engine N3 (orchestrateur)
- *Cible* : Le pipeline `assembleCompositeContext` dans engine
- *But* : Agréger les contextes de tous les boroughs en un seul workspace context
- *Parcours utilisateur* :
  . `./gradlew assembleCompositeContext -Pquery="formation FPA pipeline"`
  . L'engine appelle codebase-gradle N1 avec le contrat RAG
  . Lit `build/context/*.context.txt` de tous les boroughs
  . Produit `build/engine/composite-context.json`

=== Contrainte technique : Résilience pgvector

Le pipeline ne doit pas planter si pgvector est indisponible. La stratégie :

. `try { store.initSchema() } catch { log.warn }` → continue
. Si pgvector down → `indexBoroughSources()` skipé
. `buildScoped()` → `assembleContext()` → `try { queryService.query() } catch { fallback message }`
. Résultat : EAGER + Graphify OK, RAG = "[RAG] pgvector indisponible"

=== Points d'attention

- *Pas de duplication* : `assembleWorkspaceContext` dans engine? Ou dans codebase-plugin? → codebase-plugin (N1) pour que les tests y soient. Engine N3 l'appelle comme consommateur.
- *Pas de dépendance circulaire* : codebase-plugin N1 ne peut pas importer un N2. L'agrégation cross-borough se fait en lisant les fichiers, pas en important les plugins.
- *Jackson version* : quizz-benchmark a un mismatch 2.18.2/2.19.1. À fixer avant d'y appliquer codebase-plugin.
