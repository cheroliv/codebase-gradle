== Atelier 1 : Vision

=== QQOQCP

[cols="2,5"]
|===
| Qui | Tous les boroughs du workspace (11 boroughs N0→N4). Les développeurs humains qui lisent `./gradlew tasks`. Les agents opencode qui découvrent les capacités disponibles. Les consommateurs N3 (engine) qui orchestrent le DAG.
| Quoi | Une taxonomie unifiée à 4 verbes (GENERER/COLLECTER/TRANSFORMER/DÉPLOYER) appliquée à toutes les tâches Gradle de tous les boroughs. Un format pivot explicite (AsciiDoc structuré + `metadata.json`) comme contrat d'interface entre boroughs. Une convention over configuration pour l'inférence des métadonnées depuis le filesystem. Une configuration par domaine (namespace par borough).
| Où | Dans chaque `build.gradle.kts` de chaque plugin borough — renommage des tâches + groupes Gradle. Dans `codebase-plugin/src/main/kotlin/codebase/` — implémentation du format pivot (`Metadata.kt`, `MetadataValidator.kt`). Dans chaque borough producteur — écriture de `metadata.json` en sortie.
| Quand | *Immédiatement* — avant toute nouvelle construction de fonctionnalité. EPIC K est un prérequis de lisibilité pour la Phase 0. Sans lui, on accumule des tâches mal nommées dont le coût de renommage futur est exponentiel.
| Comment | 5 phases (K-1→K-5) : renommage des tâches → format pivot → configuration par domaine → convention over configuration → documentation. Chaque phase est un sprint indépendant de ~1-3j, exécuté par des agents @xp en parallèle sur les boroughs concernés.
| Pourquoi | Actuellement, `./gradlew tasks` liste des noms opaques : `trainingRetrieve`, `assembleCompositeContext`, `codexPipeline` — la charge cognitive est élevée, personne ne sait dans quel ordre appeler les tâches, et le contrat inter-borough est implicite (le consommateur "devine" le format de la sortie). Avec la taxonomie unifiée, le cycle de vie est lisible en une commande (`./gradlew tasks --group=generate`), et chaque consommateur vérifie le contrat `metadata.json` avant traitement.
|===

=== Product Vision Board

==== Target Group
. *Développeur humain* — lit `./gradlew tasks` pour découvrir le pipeline de formation
. *Agent opencode* — découvre les capacités disponibles par groupe Gradle
. *Engine N3* — orchestre le DAG en vérifiant les contrats `metadata.json`
. *Nouveau borough (futur)* — s'intègre en respectant le contrat sans ambiguïté

==== Needs (5 besoins principaux)
. *Lisibilité immédiate* : comprendre le cycle de vie formation en lisant `./gradlew tasks --group=collect` puis `--group=generate`
. *Contrat explicite* : chaque borough consommateur vérifie `metadata.json` avant de traiter un artefact — zéro devinette sur le format
. *DAG vérifiable* : le champ `dependencies[]` du `metadata.json` permet à engine de valider `verifyDagAcyclic` sur les dépendances déclarées
. *Onboarding nul* : un nouveau développeur comprend le flux `collect → transform → generate → deploy` sans lire 4000 lignes de doc
. *Interopérabilité outil* : AsciiDoc + JSON = lisible par n'importe quel agent, humain ou LLM

==== Product (proposition de valeur)
Une ontologie commune à 11 boroughs qui rend le DAG N0→N4 lisible,
vérifiable et extensible — sans casser l'existant.

==== 5 fonctionnalités principales
. Convention de nommage — renommer toutes les tâches Gradle selon GENERER/COLLECTER/TRANSFORMER/DÉPLOYER
. Format pivot — `metadata.json` obligatoire en sortie de tout borough producteur N2
. Configuration par domaine — chaque borough expose son extension sous un namespace dédié
. Convention over configuration — inférence des métadonnées depuis la structure du filesystem
. Documentation — mise à jour des `.agents/INDEX.adoc` et `AGENTS.adoc` workspace

==== Business Goal (critères de succès)
- 100% des tâches Gradle N2 renommées selon la taxonomie (0 tâche orpheline)
- 100% des boroughs producteurs N2 émettent un `metadata.json` valide
- `./gradlew tasks --group=generate` liste toutes les tâches de génération
- Zéro régression : tous les tests existants passent après renommage
- CI/CD vert sur tous les boroughs après application de la taxonomie

==== Vision (1 phrase)
Une ontologie à 4 verbes qui transforme 11 boroughs disparates en un pipeline lisible,
où chaque artefact porte son contrat, et où `./gradlew tasks` raconte le flux de formation.

== Atelier 2 : Personas / Parcours utilisateurs

=== Persona 1 : Développeur humain — découverte du pipeline

- *Cible* : Dev qui clone le workspace et veut comprendre le flux de formation
- *But* : Comprendre l'ordre des tâches sans lire la documentation exhaustive
- *Parcours utilisateur* :
  . `cd workspace && ./gradlew tasks`
  . Voit les groupes : `collect`, `generate`, `transform`, `deploy`
  . `./gradlew tasks --group=collect` → `collectFromCorpus` (Newark), `collectContext` (engine)
  . `./gradlew tasks --group=generate` → `generateSPG` (Manhattan), `generateSlides` (Brooklyn), `generateQuiz` (Bronx)
  . Comprend le flux en < 30 secondes

=== Persona 2 : Agent opencode — session dans un borough

- *Cible* : Agent opencode qui démarre une session de travail dans un borough N2
- *But* : Découvrir les tâches disponibles et leurs contrats
- *Parcours utilisateur* :
  . L'agent lit le `metadata.json` des artefacts d'entrée
  . Vérifie `type`, `version`, `dependencies[]`
  . Si contrat OK → exécute la tâche appropriée (ex: `transformCorpusToPdf`)
  . Si contrat KO → lève une erreur explicite ("version 2.0 non supportée, attendu 1.0")

=== Persona 3 : Engine N3 — orchestration du DAG

- *Cible* : Pipeline `assembleCompositeContext` dans engine
- *But* : Vérifier le DAG inter-boroughs avant exécution
- *Parcours utilisateur* :
  . Engine scanne tous les `metadata.json` des boroughs producteurs
  . Extrait les `dependencies[]` de chaque artefact
  . Vérifie `verifyDagAcyclic` sur le graphe de dépendances déclaré
  . Si acyclique → exécute le pipeline dans l'ordre topologique
  . Si cyclique → refuse et log l'erreur

=== Persona 4 : Nouveau borough — intégration future

- *Cible* : Développeur qui crée un nouveau plugin N2 (ex: `confluence-gradle`)
- *But* : S'intégrer dans l'écosystème sans ambiguïté
- *Parcours utilisateur* :
  . Lit `TAXONOMIE_WORKSPACE.adoc` — connaît les 4 verbes
  . Nomme ses tâches : `collectFromConfluence` (COLLECTER), `deployToConfluence` (DÉPLOYER)
  . Émet un `metadata.json` avec `source`, `type`, `version`, `dependencies`
  . S'intègre sans friction — engine le découvre automatiquement

=== User Stories détaillées

==== US-K01 — Convention de nommage sur 4 boroughs P0

*En tant que* développeur,
*je veux* renommer les tâches de Manhattan, Newark, Brooklyn et Bronx selon la taxonomie GENERER/COLLECTER/TRANSFORMER/DÉPLOYER,
*afin que* les 4 boroughs cœur du MVP0 soient lisibles en une commande `./gradlew tasks --group=generate`.

[cols="2,5"]
|===
| Sous-tâche | Détail
| Manhattan | `plannerGenerateSPG` → `generateSPG`. Ajouter `group = "generate"`.
| Newark | `trainingRetrieve` → `collectFromCorpus`. Ajouter `group = "collect"`. +
`assembleCompositeContext` → `collectContext`. Ajouter `group = "collect"`.
| Brooklyn | `codexPipeline` → `transformCorpusToPdf`. Ajouter `group = "transform"`. +
Nouvelles tâches : `generateSlides` (group=generate), `generatePdf` (group=generate).
| Bronx | `quizGenerate` → `generateQuiz`. Ajouter `group = "generate"`. +
`quizReport` → `generateQuizReport`. Ajouter `group = "generate"`.
|===

*Contrainte* : Dépréciation transitoire. L'ancien nom reste disponible avec un warning pendant 1 cycle.

*Critère d'acceptation* :
- `./gradlew tasks --group=generate` liste `generateSPG`, `generateSlides`, `generatePdf`, `generateQuiz`
- Tous les tests existants passent
- L'ancien nom affiche `[DEPRECATED] use generateSPG instead`

==== US-K02 — Classes Metadata + MetadataValidator

*En tant que* développeur,
*je veux* des classes `Metadata.kt` et `MetadataValidator.kt` dans codebase-plugin (N1),
*afin que* tous les boroughs N2 puissent les utiliser pour émettre et vérifier le contrat `metadata.json`.

[cols="2,5"]
|===
| Classe | Responsabilité
| `Metadata.kt` | Data class sealed avec champs obligatoires (`source`, `type`, `version`, `generatedAt`, `model`, `dependencies`). Sous-types par `type` : `SPGMetadata`, `SPDMetadata`, `QuizMetadata`, `PlanMetadata`. Sérialisation Jackson.
| `MetadataValidator.kt` | Vérifie la présence du fichier, la compatibilité de version (semver), l'intégrité du champ `dependencies[]`.
|===

*Critère d'acceptation* :
- `jacksonObjectMapper().readValue<Metadata>(json)` parse correctement
- `MetadataValidator.validate(file)` retourne `Result<Metadata>`
- Tests JUnit5 : JSON valide → OK, JSON invalide → erreur explicite, version mismatch → erreur

==== US-K03 — metadata.json pilote — Manhattan generateSPG

*En tant que* développeur,
*je veux* que la tâche `generateSPG` de Manhattan écrive un `metadata.json` à côté de sa sortie AsciiDoc,
*afin de* valider le contrat sur un borough pilote avant déploiement général.

*Critère d'acceptation* :
- Après `./gradlew generateSPG -Pintention="..."`, le répertoire de sortie contient `spg.adoc` et `metadata.json`
- `metadata.json` contient `"type": "SPG"`, `"source": "manhattan"`, `"version": "1.0"`
- `MetadataValidator.validate("build/spg/metadata.json")` retourne `Success`

==== US-K04 — metadata.json vérifié par engine

*En tant que* développeur,
*je veux* que le pipeline `assembleCompositeContext` de engine vérifie le `metadata.json` avant de consommer un artefact,
*afin d'* éviter les erreurs silencieuses quand un borough change son format de sortie.

*Critère d'acceptation* :
- Engine lit `foundry/public/manhattan/build/spg/metadata.json`
- Si absent → erreur explicite "metadata.json manquant pour Manhattan"
- Si `version` incompatible → erreur "version 2.0 non supportée par engine v1.x"
- Si OK → poursuit l'assemblage

==== US-K05 — Configuration par domaine pilote

*En tant que* développeur,
*je veux* que Manhattan et Newark exposent leur configuration sous des namespaces dédiés,
*afin que* les consommateurs n'aient pas à deviner les propriétés globales.

[source,kotlin]
----
// Exemple — build.gradle.kts

    ollamaUrl.set("http://localhost:11434")

training {
    corpusPath.set("data/corpus/afnor/")
    outputFormat.set("asciidoc")
}
----

*Critère d'acceptation* :
- `project.extensions.getByType(PlannerExtension::class.java)` fonctionnel
- Les propriétés sont typées (pas de `String` partout)
- Documentation dans le `README.adoc` du borough

==== US-K06 — Convention over configuration (inférence filesystem)

*En tant que* développeur,
*je veux* que le pipeline SPG→SPD infère les métadonnées de session depuis la structure du filesystem,
*afin que* l'ajout d'une nouvelle session se fasse par simple création de fichier, sans modifier une configuration centrale.

[source]
----
data/formations/
├── 01_accueil/
│   ├── 001_bienvenue.adoc     → module=accueil, session=001, ordre=1
│   └── 002_objectifs.adoc     → module=accueil, session=002, ordre=2
└── 02_core/
    ├── 003_algorithmes.adoc   → module=core, session=003, ordre=3
    └── index.adoc             → ordre=-987654321 (magic value, toujours premier)
----

*Critère d'acceptation* :
- `FileSystemMetadataExtractor.extract(File)` retourne `{module, ordre, statut, titre}`
- Les fichiers préfixés `_` sont `draft`
- `index.adoc` a l'ordre magic value `-987654321`
- Tests JUnit5 sur une arborescence fixture

==== US-K07 — Format pivot sur tous les boroughs restants

=== Contrainte technique : Dépréciation transitoire

Le renommage des tâches est un *breaking change* pour tout script, CI, et doc
qui référence l'ancien nom. La stratégie :

. *Alias déprécié* — l'ancien nom reste disponible 1 cycle avec un warning :
+
[source,kotlin]
----
project.tasks.register("trainingRetrieve") {
    group = "deprecated"
    description = "DEPRECATED: use collectFromCorpus instead"
    dependsOn("collectFromCorpus")
    doLast { logger.warn("[DEPRECATED] trainingRetrieve → use collectFromCorpus") }
}
----

. *CI/CD* — mettre à jour les workflows GitHub Actions *après* validation de l'alias
. *Documentation* — les `.agents/INDEX.adoc` référencent le nouveau nom,
  avec une note "anciennement trainingRetrieve" en commentaire

=== Contrainte technique : Compatibilité ascendante du metadata.json

Le champ `version` suit le semver. Règle :

- *MAJOR (1.0 → 2.0)* : breaking change, le consommateur DOIT être migré
- *MINOR (1.0 → 1.1)* : ajout de champs optionnels, rétrocompatible
- *PATCH (1.0.0 → 1.0.1)* : correction, transparent

Le `MetadataValidator` vérifie la compatibilité :
[source,kotlin]
----
fun isCompatible(provided: String, expected: String): Boolean {
    val (pMajor) = provided.split(".").map { it.toInt() }
    val (eMajor) = expected.split(".").map { it.toInt() }
    return pMajor == eMajor // même majeure = compatible
}
----

=== Points d'attention

- *Pas de dépendance circulaire* : Metadata est dans codebase-plugin (N1). Les N2 l'utilisent. Engine N3 lit les fichiers, n'importe pas les plugins N2.
- *Langgraph4j est prêt* : le `PlanningState` (classification, planJson, plan) est directement sérialisable en `PlanMetadata`. Ajout d'une méthode `toMetadata()` dans `FormatNode`.
- *Jackson version* : quizz-benchmark a un mismatch 2.18.2/2.19.1. À fixer avant K-2 (Metadata utilise Jackson).
- *Ordre K vs G* : EPIC K (ontologie) et EPIC G (migration CCCP) sont parallélisables. Mais K doit être fait *avant* la republication massive (EPIC G-06) pour ne pas publier sous l'ancienne taxonomie.
