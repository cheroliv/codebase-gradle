== EPIC 3 : Architecture Multi-Canal Convergent — Formalisation

**Statut** : A FAIRE
**Priorite** : P0

*Pourquoi* : Le contexte fourni au LLM ne doit pas être un bloc monolithique. Il est
composé de canaux convergents, chacun spécialisé dans un type d'information. La
convergence produit un **vecteur composite de contexte**.

=== Les Quatre Canaux

[cols="1,2,3,3"]
|===
| Canal | Technologie | Type d'information | Rôle
| **EAGER/LAZY** | Fichiers AsciiDoc + `.agents/` | Règles, procédures, backlog, historique | Gouvernance déterministe — ce que le LLM *doit* savoir
| **RAG pgvector** | LangChain4j + ONNX + PostgreSQL | Documents, code source, SPG, articles | Similarité vectorielle — ce qui est *proche* sémantiquement
| **Graphify** | `graph.json` + plantuml-gradle | Nœuds, arêtes, communautés, dépendances | Relations exactes — ce qui est *structurellement* lié
| **Ressources** | Dossiers encyclopédies, archives, datasets | Données brutes nettoyées, corpus, historique long | Mémoire froide — ce qui est *archivé* mais accessible
|===

=== Pipeline de Convergence

[source]
----
REQUÊTE UTILISATEUR
    ↓
[EAGER/LAZY] → règles applicables, backlog priorisé (~500 lignes max)
    ↓
[RAG pgvector] → top-K documents similaires (filtrés par anonymisation EPIC 2)
    ↓
[Graphify] → graphe de voisinage (arêtes exactes vers nœuds pertinents)
    ↓
[Ressources] → datasets, archives, encyclopédies (si pertinent, lazy load)
    ↓
VECTEUR COMPOSITE → LLM (deepseek-v4-pro) → réponse
----

== EPIC 13 : LangGraph4j Classifieur Vision/Opinion — Filtre Avant Cascade de Dilution

**Statut** : A FAIRE
**Priorite** : P1

*Pourquoi* : Tout ce qui est ecrit dans `office/` et `configuration/` n'est pas diluable.
Une decision architecturale validee (= VISION) doit migrer vers les documents racine.
Une strategie speculative ou une intuition (= OPINION) doit rester confinee. La grille
RGPD (EPIC 2) dit *ou* router — la classification Vision/Opinion dit *quoi* router.

Le probleme : un LLM monolithique ne peut pas distinguer fiablement Vision d'Opinion
sans filet de securite. LangGraph4j resout ca avec un graphe d'etat ou le LLM propose
et l'humain valide les cas ambigus.

**Analyse de la doc LangGraph4j** : lecture du README (`langgraph4j/langgraph4j`,
v1.8.14, 1.6k stars) valide 4 capacites cles qui repondent exactement au probleme :

* *Conditional Edges* — routage dynamique Vision→dilution / Opinion→confinement
  en fonction du statut classifie
* *Human-in-the-Loop* — pattern documente explicitement ("Implementing Human-in-the-Loop
  at ease"). Le LLM propose, l'humain tranche sur les sections ambigues.
* *Checkpoints* — sauvegarde de l'etat a chaque etape. Traçabilite complete :
  qui a classifie quoi, quand, avec quel outcome. Permet la maturation
  (une section OPINION peut evoluer en VISION — le checkpoint change).
* *Subgraphs* — le graphe de classification est un sous-graphe reutilisable,
  insere comme filtre avant la cascade de dilution (EPIC 10).

*Depend de* : EPIC 11 (LangGraph4j infrastructure — le classifieur Vision/Opinion
est un graphe specialise dans l'ecosysteme LangGraph4j)

=== Sous-taches

* [ ] Definir la data class `EpistemicStatus` : VISION, OPINION, HYPOTHESE, STRATEGIE
* [ ] Definir la data class `ClassificationState` : content (section AsciiDoc brute),
  proposedStatus (propose par le LLM), confidence (CONFIANT/INCERTAIN), approved (humain),
  finalStatus
* [ ] Implementer `ClassifierNode` : deepseek-v4-pro lit la section, propose un statut
  et un niveau de confiance
  - Patterns de detection VISION : langage declaratif ("On utilise", "L'architecture est"),
    references a des tests/sessions, decisions actees
  - Patterns de detection OPINION : langage hypothetique ("On pourrait", "Il semble que"),
    speculation sur le futur, intuitions sans ancrage experimental
  - Patterns de detection STRATEGIE : pricing, marche cible, positionnement concurrentiel
* [ ] Implementer `DilutionRouteNode` : conditional edge selon le statut final
  - VISION → cascade dilution complete (documents racine + blog si EPIC 12)
  - STRATEGIE → dilution documents racine seulement (pas de blog public)
  - OPINION / HYPOTHESE → confinement dans office/ (documente dans le rapport)
* [ ] Implementer `HumanApprovalNode` : si confiance == INCERTAIN, presente la section
  et le statut propose a l'humain. Retour possible : APPROVED / REJECTED / RECLASSIFY
* [ ] Implanter le `ClassificationGraph` comme un `StateGraph<ClassificationState>` :
  [ClassifierNode] → conditional edge (CONFIANT → DilutionRouteNode, INCERTAIN → HumanApprovalNode)
* [ ] Wiring avec EPIC 10 : inserer `ClassificationGraph` comme sous-graphe (subgraph)
  avant `DilutionExecutor`. Le filtre est optionnel, parametre `--with-classification`
* [ ] Checkpoints : chaque classification produit un checkpoint dans `.agents/`
  pour audit et maturation future
* [ ] Tests : 20 sections synthetiques (10 VISION, 5 OPINION, 5 STRATEGIE) → mesure
  du taux de classification correcte avec et sans validation humaine
